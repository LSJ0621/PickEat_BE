## 목적/범위
- 목적: 프롬프트 품질을 높이면서 토큰 비용을 줄이고, 응답을 “프론트 가공 가능한 JSON”으로 고정한다.
- 범위: 메뉴 추천 / 취향 분석 / 구글 장소 추천 프롬프트 + 응답 스키마(Structured Outputs).

## 공통 프롬프트 원칙(필수)
### 입력(컨텍스트) 구성
- 입력은 반드시 “섹션 + 구분자”로 전달한다(누락/혼선을 줄이기 위함).
- 섹션 명은 고정한다: `USER_PROMPT`, `PREFERENCES`, `PREFERENCE_ANALYSIS`, `CANDIDATES`, `CONSTRAINTS` 등.

### 출력 규칙
- 출력은 JSON 한 개만(추가 텍스트/코드블록/설명 금지).
- `response_format: json_schema` + `strict: true`를 전제로 스키마를 단순/명확하게 유지한다.
- 길이 제한/금지 규칙을 프롬프트에 명시해 장문/메타 출력을 차단한다.

### 품질/안전 가드레일
- 확실하지 않으면 “추측”하지 말고, 입력 근거 기반으로만 작성한다.
- 동일 내용 반복 금지, 과장/칭찬/심리 해석 금지.
- 금지된 필드/키 생성 금지(스키마 외 출력 금지).

## 1) 메뉴 추천 프롬프트/스키마(“추천 전체 이유 1개”)
### 요구사항 요약
- 기반 정보 3종을 모두 사용:
  1) 취향정보(likes/dislikes)
  2) AI 취향분석정보(analysis)
  3) 유저가 작성한 프롬프트(userPrompt)
- 메뉴별 reason 금지 → 추천 전체를 설명하는 `reason` 1개만 반환.

### 입력 포맷(예: User Prompt 템플릿)
아래 형식을 고정한다:
- `USER_PROMPT`: 유저 원문
- `PREFERENCES`: likes/dislikes
- `PREFERENCE_ANALYSIS`: 분석 텍스트(없으면 “없음”)
- `CONSTRAINTS`: 추천 개수/탐색형 포함 등 규칙

### 출력 스키마(안)
```json
{
  "recommendations": ["string"],
  "reason": "string"
}
```

### 출력 제약(프롬프트에 명시)
- `recommendations`: 1~5개, 표준 “메뉴명”만(괄호/영어/추가설명/합성메뉴 금지), 중복 금지.
- “탐색형 메뉴” 1개 포함(취향과 조금 다르지만 맥락상 납득 가능한 수준).
- `reason`: 500자 내외, 존댓말, 아래 3요소를 “한 번에” 설명(메뉴별 분리 금지)
  - 최근/오늘 맥락(유저 프롬프트 근거)
  - 자주 먹는 특징(취향정보/분석 근거)
  - 탐색형 메뉴를 섞은 이유

### 수용 기준(Acceptance Criteria)
- JSON만 반환한다.
- `reason`이 메뉴별로 쪼개지지 않는다.
- 추천 메뉴에 불필요한 수식/설명이 없다.

## 2) 취향 분석 프롬프트/스키마(톤/길이/일관성)
### 목표
- 존댓말 고정, 중복 최소화, 짧고 일관된 분석.

### 출력 스키마(현행 유지)
```json
{ "analysis": "string" }
```

### 출력 제약(프롬프트에 명시)
- 말투: 존댓말(종결 `~습니다/~예요`).
- 길이: 공백 포함 500자 이내(가능하면 400자 내로).
- 구조: 선호 요약(1~2문장) → 변화(1단락) → 부족 데이터(선택).
- 금지: 반말/과장/심리추측/메뉴명 장문 나열.

## 3) 구글 장소 추천(토큰 절감 + JSON 요약 근거)
### 문제/목표
- 후보/리뷰 입력이 커서 “입력 토큰”이 과다하다.
- 성능(정확도/다양성)은 유지하면서, 입력 토큰을 크게 줄인다.
- 근거는 장문 reason 대신 “특징 요약 JSON”으로 받아 프론트에서 가공한다.

### 3-1) 입력 토큰 절감(백엔드 전처리: 핵심)
- 후보 수 제한: 상위 N개만 전달(예: 8~12).
- 필드 최소화: `id, name, rating, userRatingCount, priceLevel` 중심(필요 시 `types`/`address`만 추가).
- 리뷰 최소화: 0~2개, 텍스트 80~120자 절단(메뉴 키워드 포함 리뷰 우선).
- 직렬화: pretty print 금지(공백/개행 제거), null/빈 값 제거.

### 3-2) 프롬프트 규칙(짧고 강하게)
- 후보(`CANDIDATES`) 중에서만 선택.
- 검색어(메뉴)와 관련 없는 가게는 제외(가게명/리뷰에 근거가 없으면 제외).
- 관련 가게가 없으면 빈 배열 가능.
- JSON 스키마 외 텍스트 금지.

### 3-3) 출력 스키마(안: 프론트 가공 최적화)
```json
{
  "recommendations": [
    {
      "placeId": "string",
      "name": "string",
      "reason": {
        "evidence": ["string"],
        "pros": ["string"],
        "cautions": ["string"]
      }
    }
  ]
}
```

### 출력 제약(프롬프트에 명시)
- 최대 3개 추천.
- `placeId`: 후보의 `id`를 그대로 사용(변형 금지).
- `reason.evidence/pros/cautions`: 각 0~3개, 각 항목은 짧은 구(예: 10~25자), 장문 금지.
- `max_completion_tokens`는 낮게(예: 250~450) 설정해 장문 출력을 구조적으로 제한.
