# PickEat Backend 테스트 전략 회고록

이 문서는 PickEat Backend 테스트 전략의 실제 적용 과정에서 발생한 주요 이슈와 그 해결 과정을 기록한다. 각 회고는 "증상 → 직접 원인 → 구조 원인 → 시스템 개선안 → 재발 방지 → 교훈"의 5단계 분석을 따른다.

## 목차

1. [BE E2E 45건 실패 → 0건 수정](#회고-1-be-e2e-45건-실패--0건-2026-04-11) — 타입 시스템 밖의 계약 검증

## 작성 원칙

- **사실 기반 기록**: 날짜, 수치, 소스 경로를 명시한다. 추측은 배제한다.
- **비난 금지**: 문제는 시스템에, 해결은 구조에서 찾는다.
- **재사용 가능한 교훈**: 이 프로젝트 밖에서도 통용될 수 있는 형태로 정리한다.

## 회고 1: BE E2E 45건 실패 → 0건 (2026-04-11)

### 배경

테스트 인프라 세팅을 마치고 처음으로 Backend E2E 전체를 실행했을 때, 45건의 테스트가 실패했다. 단일 원인이 아니라 그간 기능 개발을 진행하면서 누적된 테스트-구현 불일치가 한꺼번에 드러난 상황이었다.

최종적으로 소스 10개 파일 + 테스트 7개 파일을 수정하여 45 → 0 건으로 정리했다. 이 작업의 가치는 "45건을 고쳤다"가 아니라, 실패 45건이 **몇 가지 반복 가능한 패턴**으로 분류된다는 것을 확인한 데 있다.

### 패턴 분류

| 패턴 | 건수 | 대표 증상 |
|------|------|-----------|
| `@HttpCode` 누락 | 4 | `POST`가 기본 201을 반환하는데 테스트는 200 기대 |
| 응답 구조 불일치 | 3 | DTO/Response interface 변경 이후 테스트 미반영 |
| OpenAI mock 우선순위 | 1 | 전역 mock이 개별 케이스 mock을 덮어씀 |
| Error status 정책 변경 (404/403 → 400) | 2 | 에러 정책을 바꿨으나 테스트가 옛 status 유지 |
| 모듈 등록 누락 (AdminNotification) | 1 | 모듈을 만들었으나 `AppModule.imports`에 추가 안 함 |
| 개별 이슈 (Redis 캐시 leak, JWT `jti`, object spread 순서 등) | 3 | 단발성, 공통 패턴 없음 |

> 합계 14건은 "루트 원인" 단위이며, 각 루트 원인이 여러 테스트를 동시에 실패시켜 총 45건이 되었다.

### 근본 원인 분석

#### 패턴 1: `@HttpCode` 누락 (4건)

**직접 원인**
NestJS는 `@Post()`가 기본적으로 `201 Created`를 반환한다. 개발자가 "이 엔드포인트는 200을 돌려준다"고 설계했더라도, 컨트롤러에 `@HttpCode(HttpStatus.OK)`를 명시하지 않으면 실제 응답은 201로 나간다. 이 경우 `supertest`의 `.expect(200)`은 실패한다.

**구조 원인**
HTTP status code는 **TypeScript의 타입 시스템 밖에 존재하는 런타임 계약**이다. 컨트롤러의 반환 타입이 `Promise<UserResponse>`이든 무엇이든, 응답 status는 타입 검사로 잡을 수 없다. 즉, 타입이 통과해도 실제 응답이 의도와 다를 수 있다. 이는 단순한 "실수"가 아니라 **TypeScript가 원천적으로 방어할 수 없는 영역**이다.

**시스템 개선안**

| 선택지 | 장점 | 단점 |
|--------|------|------|
| ESLint custom rule: `@Post + @HttpCode` 필수 | 린트 단계에서 차단 | 사내 룰 유지 비용 |
| 공통 데코레이터 (`@OkPost()`, `@CreatedPost()`) | 의도가 코드로 드러남 | 기존 코드 일괄 전환 필요 |
| **E2E에서 status 엄격 검증** (채택) | 추가 인프라 0, 기존 테스트 강화 | 린트 단계가 아닌 테스트 단계에서 차단 |

**재발 방지**
E2E 테스트 작성 시 `.expect(200)`을 "일단 통과하면 OK"가 아니라 **의도한 정확값**으로 다룬다. 테스트 관점의 변화가 실제 해결책이다.

#### 패턴 2: 응답 구조 불일치 (3건)

**직접 원인**
Response DTO가 바뀌었으나 (예: `user.profile.nickname` → `user.nickname`), E2E 테스트는 옛 구조를 assert 하고 있었다.

**구조 원인**
DTO interface는 FE/BE 양측에서 참조되는데, E2E 테스트는 **FE와 BE 사이에 낀 3자**이다. DTO 변경 시 FE는 타입 에러로 바로 드러나지만, E2E는 런타임에 맞춰 작성되어 있어 구조 변경의 충격파를 직접 받는다.

**시스템 개선안**
E2E 테스트에서도 DTO 타입을 import 하여 assert 하도록 한다:
```typescript
const body: UserResponse = res.body;
expect(body.nickname).toBe('kim');  // 타입 시스템이 nickname 존재를 보증
```

**재발 방지**
"테스트는 문자열 키로 assert 해도 된다"는 관행을 버린다. E2E라도 타입을 끌어와 **컴파일 타임에 스키마 drift를 잡는다**.

#### 패턴 3: OpenAI mock 우선순위 (1건)

**직접 원인**
전역 mock이 `vi.mock()`으로 설정되었고, 특정 테스트에서 `mockImplementationOnce()`로 개별 응답을 주입했다. 그러나 다른 테스트의 beforeEach가 전역 mock을 `mockResolvedValue()`로 덮어쓰면서, **선언 순서에 따라 기대와 다른 응답이 반환**되었다.

**구조 원인**
Vitest mock의 "리셋 시점"과 "덮어쓰기 우선순위"가 직관적이지 않다. `mockImplementationOnce` → `mockResolvedValue` → `mockResolvedValueOnce` 조합의 동작은 문서를 보지 않으면 예측 불가능하다.

**시스템 개선안**
- 외부 SDK mock은 **팩토리 함수 하나**에서만 구성한다.
- `beforeEach`에서 `mockReset()` 후 필요한 응답을 주입하는 단일 경로를 강제한다.

**재발 방지**
"전역 mock + 개별 덮어쓰기" 혼용을 금지한다. 테스트 파일 상단에 mock helper를 두고, 모든 주입을 이 helper를 통해서만 수행한다.

#### 패턴 4: Error status 정책 변경 (2건)

**직접 원인**
BE에서 "리소스 없음"을 404 → 400으로, "권한 없음"을 403 → 400으로 통일하는 정책 변경이 있었다. 구현은 반영되었으나 일부 테스트가 옛 status를 기대하고 있었다.

**구조 원인**
정책 변경을 "에러 필터 한 곳"에서 처리한 것이 아니라 **컨트롤러 개별 throw**에 분산되어 있었다. 테스트도 마찬가지로 컨트롤러별로 흩어져 있어 일괄 반영이 누락되었다.

**시스템 개선안**
- 공통 `BusinessException` + 전역 `ExceptionFilter`를 통해 status 정책을 한 곳에서 결정한다.
- 테스트도 에러 케이스를 공통 helper로 검증 (`expectBusinessError(res, 'USER_NOT_FOUND')`).

**재발 방지**
에러 정책은 **단일 소스 오브 트루스**를 두고, 이 정책 변경 시 한 파일만 수정하도록 구조화한다.

#### 패턴 5: 모듈 등록 누락 (1건)

**직접 원인**
`AdminNotificationModule`을 생성했지만 `AppModule.imports`에 추가하지 않아 런타임에 Provider 주입 실패.

**구조 원인**
NestJS의 모듈 등록은 **수동 import** 에 의존한다. 파일 존재 ≠ 앱에 편입. 이는 컴파일러가 잡을 수 없는 또 다른 "타입 밖의 계약"이다.

**시스템 개선안**
- E2E 부팅 테스트 한 개를 항상 유지: `Test.createTestingModule({ imports: [AppModule] })`가 실패 없이 컴파일되는지 검증.
- 새 모듈 추가 시 CLI 스니펫/템플릿에 `imports` 업데이트 체크리스트 포함.

#### 패턴 6: 개별 이슈 (3건)

- **Redis 캐시 leak**: 테스트 종료 시 Redis 커넥션이 닫히지 않아 다음 테스트에 영향. `afterAll`에서 명시적 close.
- **JWT `jti` 중복**: 짧은 시간에 두 번 발급 시 동일 `jti` 생성. 발급 로직에 `crypto.randomUUID` 추가.
- **Object spread 순서**: `{ ...defaults, ...input }` 의도였으나 실제로는 `{ ...input, ...defaults }`로 작성되어 사용자 입력이 덮여 씌워짐.

세 건 모두 패턴화할 수 없는 개별 버그이나, 공통점은 **E2E가 아니었다면 발견이 늦어졌을** 사안이라는 것이다.

### 교훈

1. **타입 시스템 밖의 계약은 런타임 검증이 유일한 방어선이다.** HTTP status, 모듈 등록, 외부 SDK mock 동작 — 이들은 컴파일러가 잡을 수 없다. E2E는 이 공백을 메우는 유일한 도구다.
2. **"45건 실패"는 실패가 아니라 E2E가 역할을 다했다는 증거다.** 이 버그들이 프로덕션에 갔다면 복구 비용은 훨씬 컸을 것이다. 테스트가 실패하는 것이 테스트의 **성공**이다.
3. **패턴 분류 → 시스템 개선이 회고의 핵심이다.** "각각 고쳤다"로 끝내면 같은 실패가 또 쌓인다. "어떤 구조가 이 실패를 허용했는가"를 묻고, 구조를 바꿔야 한다.
