## Pick-Eat PRD (Product Requirements Document)

> 버전: v0  
> 대상: Pick-Eat 서버 (NestJS 기반 백엔드)  

---

## 1. 서비스 개요

- **서비스 이름**: Pick-Eat
- **한 줄 정의**:  
  사용자의 취향, 상황(기분/컨디션/날씨/장소), 과거 선택 기록을 반영해서  
  “오늘 뭐 먹지?” 고민을 최소화해 주는 **AI 메뉴 추천 서비스**.
- **핵심 가치**
  - 결정 피로 감소: 메뉴 선택 스트레스를 줄여준다.
  - 취향 존중: 사용자의 장기적인 취향 패턴을 학습하고 반영한다.
  - 맥락 반영: 오늘 기분/날씨/위치 등 그날의 맥락을 고려한다.

---

## 2. 목표 및 지표

### 2.1 비즈니스/제품 목표

- 사용자가 “오늘 뭐 먹지?” 고민할 때 가장 먼저 떠올리는 도구가 되는 것.
- **하루에 한 번 이상** Pick-Eat을 열어 추천을 확인하게 만드는 것.
- 추천 결과가 실제 선택으로 이어지는 비율을 높이는 것.

### 2.2 핵심 지표 (KPI)

- **일간 메뉴 추천 요청 수 (DAU 기반)**  
  - 사용자 1인당 하루 평균 추천 요청 횟수
- **추천 → 실제 선택 저장 비율**
  - 추천 결과 중 사용자가 “이걸 먹었다”로 선택한 비율
- **재방문율**
  - 최근 7일 내 최소 2회 이상 추천을 요청한 사용자 비율

---

## 3. 대상 사용자 & 페르소나

### 3.1 페르소나 A – 직장인 점심/저녁 난민

- 20–40대 직장인, 회사 근처/집 근처에서 자주 시켜 먹거나 외식.
- 특징:
  - “매일 메뉴 고르는 게 귀찮다.”
  - “같은 메뉴만 계속 먹는 건 싫다.”
  - “기분/컨디션 따라 먹고 싶은 게 달라진다.”

### 3.2 페르소나 B – 선택장애 있는 커플/친구 모임

- 둘 이상이 함께 먹을 메뉴를 정해야 하는 상황.
- 특징:
  - 서로 취향이 달라서 메뉴 정하기가 어려움.
  - “이거 먹자고 했다가 욕 먹기 싫다.”

> v0에서는 **1인 기준 추천**을 우선 구현하고, 다인 추천은 이후 확장 대상으로 둔다.

## 4. 핵심 시나리오

### 4.1 첫 가입 & 취향 입력

1. 사용자가 소셜 로그인 또는 이메일로 가입한다.
2. 가입 과정에서 **음식 취향/제약/패턴**을 자유롭게 입력하거나, 템플릿 질문에 답변한다.
   - 예: “매운 음식 좋아함”, “같은 메뉴 반복 싫어함”, “야식은 가볍게 먹고 싶음”.
3. 서버는 이 정보를 구조화된 형태(태그/JSON 등)로 저장한다.

### 4.2 오늘 메뉴 추천 받기

1. 사용자가 “오늘 뭐 먹지?” 화면에서 **프롬프트**를 작성한다.
   - 예: “오늘 기분 안 좋고, 속도 좀 안 좋음”, “밖에 비 오는데 따뜻한 국물 먹고 싶어”.
2. (선택) 오늘 먹을 위치(주소)나 현재 위치를 지정한다.
3. 서버는:
   - 사용자 기본 취향 + 과거 선택/피드백 + 오늘 프롬프트 + 위치 정보를 합쳐
   - OpenAI 등 LLM을 호출해 메뉴 후보 리스트를 생성한다.
4. 클라이언트는:
   - 추천 메뉴 리스트를 보여주고,
   - 사용자가 “이걸 먹을래”를 선택할 수 있게 한다.

### 4.3 추천 이력 & 선택 이력 조회

1. 사용자는 “어제 뭐 추천 받았지?”, “최근에 뭐 먹었지?”를 확인할 수 있다.
2. 서버는:
   - 추천 이력(추천 시점, 프롬프트, 추천된 메뉴들),
   - 실제 선택 이력(어떤 메뉴를 골랐는지, 날짜/시간)을 조회할 수 있는 API를 제공한다.

### 4.4 주소/지역 관리 & 주변 식당 보기 (선택적)

1. 사용자는 자주 쓰는 주소(집/회사 등)를 최대 N개까지 등록한다.
2. 기본 주소 / 추천/검색에 사용할 주소를 지정할 수 있다.
3. 서버는:
   - 주소 기반으로 주변 식당 데이터(외부 API: Kakao/Naver/Google 등)를 조회해서 지도/리스트로 내려줄 수 있다.

## 5. 기능 요구사항 (도메인별)

### 5.1 인증/사용자 관리 (Auth/User)

- 소셜 로그인 + 이메일 로그인(선택)을 지원한다.
- 사용자 프로필:
  - 기본 정보: 닉네임, 이메일, 가입 채널, 생성일 등
  - 취향 정보: 선호/비선호 태그, 분석 결과(요약 텍스트 등)
  - 주소 정보(옵션): 기본 주소, 추천/검색용 주소

필수 요구사항:
- [ ] JWT 기반 인증 (액세스 토큰, 리프레시 토큰)
- [ ] 이메일/소셜 계정 중복 가입 방지
- [ ] 비밀번호 초기화/변경 (이메일 기반) – 필요 시

### 5.2 취향 관리 (Preferences)

기능:
- [ ] 가입 시 취향 태그/설명 입력
- [ ] 이후 언제든지 취향 수정 가능
- [ ] 취향 데이터는 구조화된 JSON 형태로 저장
  - 예: `{ "likes": [...], "dislikes": [...], "constraints": [...], "analysis": "LLM이 생성한 요약" }`
- [ ] 메뉴 추천 시 항상 이 취향 정보를 함께 사용

비고:
- 취향 입력은 자유서술 + 태그 혼합 형태를 허용하고, 서버가 이를 통합/정규화한다.

### 5.3 메뉴 추천 (Menu Recommendation)

기능:
- [ ] 사용자가 프롬프트를 보내면, LLM을 통해 **메뉴 후보 리스트**를 만들어 반환
- [ ] 각 추천에는 최소한 다음 정보 포함:
  - 메뉴 이름 (예: “마라탕”)
  - 카테고리/설명 (예: “매운 중국식 볶음 요리”)
  - 추천 이유 (선택: “매운 음식 좋아하고 오늘 비가 와서 따뜻한 국물 추천”)
- [ ] 추천은 3~10개 사이로 제한 (과도한 옵션 방지)

입력 요소:
- 사용자 취향 데이터 (likes/dislikes/constraints/analysis)
- 오늘 프롬프트 (자연어)
- 위치 정보 (주소/위도/경도, 옵션)
- 최근 선택/이력 (가능하다면)

출력 요구사항:
- [ ] 추천 메뉴 리스트
- [ ] 추천 시각 (`recommendedAt`)
- [ ] (옵션) 추천 세션 ID (추후 이력/선택과 연결)

### 5.4 추천 이력 & 선택 이력 (History & Selection)

기능:
- [ ] 추천 이력 저장
  - 사용자 ID
  - 추천 시각
  - 사용된 프롬프트
  - 추천된 메뉴 리스트 (문자열 배열 또는 구조화된 형태)
- [ ] 선택 이력 저장
  - 어떤 추천 세션에서
  - 어떤 메뉴를 실제로 골랐는지
  - 선택 시각
- [ ] 날짜/기간별로 이력 조회
- [ ] “다시 추천해줘” 기능에서 과거 이력을 재사용 가능

### 5.5 주소/위치 및 주변 식당 (Location & Places) – 확장 기능

기능:
- [ ] 사용자 주소 관리 (집/회사/기타, 최대 N개)
- [ ] 기본 주소, 추천/검색용 주소 지정
- [ ] 위치 기반 추천 옵션:
  - 추천 메뉴와 잘 맞는 주변 식당 목록 조회
  - 외부 API(Google Places, Kakao, Naver 등)를 통해 식당 데이터 수집

이 기능은 **MVP 이후 단계**에서 점진적으로 붙이는 것을 전제로 한다.

## 6. API 요구사항 (상위 설계)

> 실제 엔드포인트/스키마는 구현 단계에서 조정 가능.  
> 여기서는 “무엇을 할 수 있어야 하는지” 수준으로만 정의한다.

### 6.1 취향 관리 API (예시)

- `GET /user/preferences`
  - 현재 저장된 취향 정보 조회
- `POST /user/preferences`
  - 최초 취향 입력 또는 전체 수정

### 6.2 메뉴 추천 API

- `POST /menu/recommend`
  - Body:
    - `prompt`: string – 오늘의 상황/기분/요청
    - (옵션) `location`: { lat, lng } 또는 `address`: string
  - Response:
    - `recommendations`: 메뉴 리스트
    - `recommendedAt`
    - `recommendationId`: 추천 세션 ID

### 6.3 추천 이력 & 선택 이력 API

- `GET /menu/recommendations/history?date=YYYY-MM-DD`
  - 해당 날짜 또는 최근 N일 추천 이력 조회
- `POST /menu/selections`
  - 사용자가 실제로 선택한 메뉴 저장

### 6.4 주소/위치 API (선택)

- `GET /user/addresses`
- `POST /user/addresses`
- `PATCH /user/addresses/:id`
- `DELETE /user/addresses/:id`
- `PATCH /user/addresses/:id/default`
- `PATCH /user/addresses/:id/search`

## 7. 데이터 모델 (개념 수준)

> 실제 엔티티/테이블 설계는 이 개념 모델을 기반으로 자유롭게 조정 가능.

- **User**
  - id, email, nickname, authType, createdAt, updatedAt, deletedAt?
- **UserPreferences**
  - userId
  - likes[], dislikes[], constraints[]
  - analysis (LLM이 생성한 취향 요약)
- **UserAddress**
  - id, userId
  - roadAddress, latitude, longitude
  - isDefault, isSearchAddress
  - alias (집/회사 등)
- **MenuRecommendation**
  - id, userId
  - prompt
  - recommendations[] (문자열 또는 구조화된 메뉴 객체 배열)
  - recommendedAt
  - requestAddress / requestLocation(옵션)
- **MenuSelection**
  - id, userId
  - recommendationId
  - selectedMenus[] 또는 selectedMenu
  - selectedAt

## 8. 비기능 요구사항

### 8.1 성능 & 확장성

- 메뉴 추천 API:
  - LLM 호출 특성상 **수 초 이내 응답**(3–8초)을 목표로 한다.
  - 과도한 동시 호출 방지를 위해 사용자/토큰 단위 rate limit 고려.

### 8.2 안정성 & 로깅

- 모든 LLM/외부 API 호출에 대해:
  - 요청 요약, 응답 상태, 소요 시간, 토큰 사용량(LLM)을 로그에 남긴다.
  - 에러 발생 시 원인 파악이 가능하도록 충분한 정보를 남기되, 개인정보/비밀키는 마스킹.

### 8.3 보안

- JWT 기반 인증, HTTPS 전제.
- 사용자별 데이터 격리:
  - 다른 사용자의 추천/이력을 조회할 수 없어야 한다.

### 8.4 비용 관리

- LLM 호출/외부 API 호출 수를 모니터링하고,  
  사용량/비용을 주기적으로 확인할 수 있는 통계를 쌓는다.
