# 비밀번호 재설정 기능 구현

## 개요
사용자가 비밀번호를 잊었을 경우, 이메일 인증을 통해 비밀번호를 재설정할 수 있는 기능을 구현합니다.

## 요구사항

### 1. 기능 흐름
1. 로그인 화면에서 "비밀번호를 잊으셨나요?" 옆에 "비밀번호 재설정" 버튼 클릭
2. 이메일 입력 화면 표시
3. 이메일 입력 후 인증번호 발송 요청 (`POST /auth/password/reset/send-code`)
4. 인증번호 입력 후 검증 요청 (`POST /auth/password/reset/verify-code`)
5. 인증 성공 시 비밀번호 재설정 화면 표시
6. 새 비밀번호 입력 후 재설정 요청 (`POST /auth/password/reset`)

### 2. API 엔드포인트

#### 2.1 인증번호 발송
- **엔드포인트**: `POST /auth/password/reset/send-code`
- **요청 본문**:
  ```json
  {
    "email": "user@example.com"
  }
  ```
- **응답**:
  ```json
  {
    "success": true,
    "message": "인증번호가 발송되었습니다. 남은 재발송 횟수는 4회입니다.",
    "remainCount": 4
  }
  ```
- **처리 순서 (반드시 순서대로 진행)**:
  1. **먼저 사용자 존재 여부 확인** (User 테이블 조회)
     - 사용자가 없으면 → 즉시 에러 반환 (이메일 인증 진행하지 않음)
  2. **소셜 로그인 계정 여부 확인** (password가 null인지 확인)
     - 소셜 로그인 계정이면 → 즉시 에러 반환 (이메일 인증 진행하지 않음)
  3. **일반 회원 계정인 경우에만** → 인증번호 발송 진행
     - 위 두 단계를 통과한 경우에만 `EmailVerificationService.sendCode` 호출
- **에러 케이스**:
  - 이메일이 존재하지 않는 경우: `400 Bad Request` - "등록되지 않은 이메일입니다."
  - 소셜 로그인 계정인 경우: `400 Bad Request` - "소셜 로그인으로 가입한 계정입니다."
  - 기존 이메일 인증 시스템의 제한사항 적용 (하루 5회, 30초 쿨타임 등)

#### 2.2 인증번호 검증
- **엔드포인트**: `POST /auth/password/reset/verify-code`
- **요청 본문**:
  ```json
  {
    "email": "user@example.com",
    "code": "123456"
  }
  ```
- **응답**: `{ "success": true }`
- **주의사항**: `EmailPurpose.RESET_PASSWORD` 목적의 인증번호만 검증
- **에러 케이스**: 기존 이메일 인증 시스템의 에러 메시지와 동일

#### 2.3 비밀번호 재설정
- **엔드포인트**: `POST /auth/password/reset`
- **요청 본문**:
  ```json
  {
    "email": "user@example.com",
    "newPassword": "newPassword123!"
  }
  ```
- **응답**: `{ "success": true, "message": "비밀번호가 성공적으로 변경되었습니다." }`
- **에러 케이스**:
 - 이메일 인증이 완료되지 않은 경우: `400 Bad Request` - "이메일 인증이 완료되지 않았습니다."
 - 이메일이 존재하지 않는 경우: `400 Bad Request` - "등록되지 않은 이메일입니다."
 - 소셜 로그인 계정인 경우: `400 Bad Request` - "소셜 로그인으로 가입한 계정입니다."
 - 비밀번호 형식이 올바르지 않은 경우: `400 Bad Request` - "비밀번호 형식이 올바르지 않습니다."
  - 일일 변경 한도 초과: `429 Too Many Requests` - "비밀번호는 하루에 한 번만 변경할 수 있습니다."
- **인증 소모 처리**: 비밀번호가 변경되면 해당 이메일의 `RESET_PASSWORD` 인증 기록을 만료시켜 재사용을 막고, 다시 재설정하려면 새 코드를 받아야 함 (DB 삭제 대신 만료 처리)

### 3. 구현 세부사항

#### 3.1 기존 시스템 활용
- `EmailVerificationService`의 `sendCode`, `verifyCode`, `isEmailVerified` 메서드 활용
- `EmailPurpose.RESET_PASSWORD` 사용
- 기존 이메일 인증 템플릿 재사용하되 purpose에 따라 copy/subject를 분기해 비밀번호 재설정임을 명확히 노출

#### 3.2 인증번호 발송 로직
**⚠️ 중요: 사용자 존재 여부를 먼저 확인한 후에만 이메일 인증을 진행해야 합니다.**

1. **먼저 사용자 존재 여부 확인** (User 테이블에서 조회)
   - `userService.findByEmail(email)` 호출
   - 사용자가 없으면 → 즉시 "등록되지 않은 이메일입니다." 에러 반환 (이메일 인증 진행하지 않음)
   
2. **소셜 로그인 계정 여부 확인** (password가 null인지 확인)
   - `user.password === null`이면 → "소셜 로그인으로 가입한 계정입니다." 에러 반환 (이메일 인증 진행하지 않음)
   
3. **일반 회원 계정인 경우에만** → `EmailVerificationService.sendCode` 호출 (purpose: RESET_PASSWORD)
   - 사용자 존재 여부와 계정 타입 확인이 완료된 후에만 인증번호 발송

#### 3.3 비밀번호 재설정 로직
1. 이메일 인증 완료 여부 확인 (`isEmailVerified` with `RESET_PASSWORD`; 인증 완료 플래그로만 판정, 추가 코드 입력 없음)
2. 사용자 존재 여부 확인 (User 테이블)
3. 소셜 로그인 계정 여부 확인 (password가 null이면 에러)
4. 새 비밀번호 해싱 (bcrypt)
5. 비밀번호 업데이트
6. 해당 이메일의 `RESET_PASSWORD` 인증 기록을 만료 처리해 재사용 불가하도록 함 (다음 재설정 시 새 코드 요청 필요)
7. 비밀번호 변경 일일 1회 제한 로직 추가 (마지막 변경 시각을 저장/조회하여 24시간 내 추가 변경 요청 시 429 반환)

#### 3.4 보안 고려사항
- 인증번호는 3분간만 유효
- 인증 완료 후 비밀번호 재설정까지 일정 시간 제한 (예: 10분) 고려
- 비밀번호 재설정 후 기존 세션 무효화 (선택사항)
- 비밀번호 재설정 완료 시 이메일 알림 발송 (선택사항)
- 인증번호 재발송 일일 횟수 제한: RESET_PASSWORD 목적 기준 하루 5회까지만 발송 (초과 시 429 반환)

### 6. 에러 메시지 정리
| 상황 | HTTP 상태 | 메시지 |
|------|----------|--------|
| 이메일 미등록 | 400 | "등록되지 않은 이메일입니다." |
| 소셜 로그인 계정 | 400 | "소셜 로그인으로 가입한 계정입니다." |
| 인증 미완료 | 400 | "이메일 인증이 완료되지 않았습니다." |
| 비밀번호 형식 오류 | 400 | "비밀번호 형식이 올바르지 않습니다." |
| 인증번호 발송 제한 (쿨타임/일일 5회) | 429 | 기존 이메일 인증 시스템 메시지 |
| 인증번호 검증 실패 | 400 | 기존 이메일 인증 시스템 메시지 |
| 비밀번호 변경 일일 한도 초과 | 429 | "비밀번호는 하루에 한 번만 변경할 수 있습니다." |

### 7. 프론트엔드 연동 가이드
1. 로그인 화면에 "비밀번호 재설정" 버튼 추가
2. 이메일 입력 → `POST /auth/password/reset/send-code` (body: `{ "email": "<user email>" }`, purpose는 서버에서 RESET_PASSWORD로 처리)
3. 인증번호 입력 → `POST /auth/password/reset/verify-code` (body: `{ "email": "<user email>", "code": "<6-digit>" }`)
4. 인증 성공 시 비밀번호 재설정 화면으로 이동
5. 새 비밀번호 입력 → `POST /auth/password/reset` (body: `{ "email": "<user email>", "newPassword": "<new pwd>" }`; 인증 코드는 다시 보내지 않음)
6. 성공 응답을 받으면 이전 인증은 만료되므로, 추가로 비밀번호를 바꾸려면 다시 2~5 단계를 진행해야 함
7. 성공 시 로그인 화면으로 리다이렉트
8. 모든 요청은 `RESET_PASSWORD` purpose로 동작해야 하므로, API 호출 시 프론트에서 별도 query/body 파라미터를 추가할 필요는 없지만, 인증번호 입력 화면은 반드시 비밀번호 재설정 플로우 전용으로 노출 (회원가입 인증 화면 재사용 시 copy/색상 등으로 구분)
9. 각 화면에서 실패 응답 메시지를 그대로 노출하고, HTTP 400일 때는 입력값 초기화 없이 에러 텍스트만 표시 (다시 입력 가능), 401/403/500 등 비정상 오류는 공통 토스트/다이얼로그로 처리
10. 인증번호 발송 시 쿨타임/횟수 제한 메시지를 그대로 보여주고, 서버에서 반환한 `retryAfter`나 `remainCount`가 있으면 버튼 비활성화/남은 횟수 표기 등을 선택적으로 적용 (하루 5회 초과 시 429 처리). 응답의 `message`가 "남은 재발송 횟수는 X회입니다." 형태이므로 그대로 노출하고, `remainCount`로 버튼 텍스트/보조 문구를 업데이트하거나 `remainCount === 0`이면 버튼을 비활성화.
11. 비밀번호 변경 일일 1회 제한에 걸리면 서버가 429와 에러 메시지를 반환하므로, 비밀번호 재설정 화면에서 토스트/다이얼로그로 알리고 추가 입력은 그대로 유지하여 익일 다시 시도하도록 안내
12. 인증 성공 후 비밀번호 재설정 화면으로 이동할 때, 프론트 상태(예: recoil/zustand/store)에 `email`을 유지해 다시 입력하지 않도록 하고, 새로고침 시 플로우가 끊기는 것을 방지하려면 이메일을 쿼리스트링 또는 세션스토리지에 저장했다가 성공/실패 시 삭제
13. 비밀번호 재설정 요청 시 서버에서 비밀번호 형식 오류를 반환하면 해당 필드에 inline 에러로 표시하고, 규칙(길이/특수문자 포함 등)을 프런트에서도 동일하게 검증해 사전 차단
14. 성공 후에는 로컬에 캐싱된 인증 관련 상태(email, code 입력값, 타이머 등)를 모두 초기화한 뒤 로그인 화면으로 이동하거나 로그인 모달을 자동 오픈하는 UX를 고려
